# Бенчмарк для ВКР
Лёгкий self-contained бенчмарк для Linux-одноплатников.

Его цель — не заменить специализированные инструменты вроде `fio`, `cyclictest` или `iperf3`, а дать **один переносимый бинарник на C**, который позволяет сравнивать одноплатники по длительной стабильности, тепловому режиму, tail latency, pressure stalls и таймерному jitter.

- имя сценария сохраняется в каждом запуске
- `summary.json` содержит производные метрики для сравнения
- CPU:
  - пиковый throughput
  - sustained throughput (вторая половина прогона)
  - drift в процентах
  - время до первого падения на 5%
  - время до первого падения на 10%
  - время до троттлинга
- Thermal:
  - начальная температура
  - конечная температура
  - максимальная температура
  - скорость роста температуры в °C/мин
- Storage:
  - p50/p95/p99
  - сравнение `fsync` p99 в первой и второй половине прогона
  - рост `fsync` p99 во времени
  - metadata p99
- Realtime:
  - p50/p95/p99/max jitter
  - количество пробуждений выше 500 мкс и 1000 мкс
- `system_info.json` с системным контекстом запуска
- после каждого прогона программа возвращается в меню
- `q` завершает приложение
- `Ctrl+C` останавливает только текущий прогон и возвращает в меню

## Сборка

```bash
gcc -std=c11 -O2 -pthread sbc_bench_v3.c -lm -o sbc_bench_v3
```

## Запуск

```bash
./sbc_bench_v3
```

## Команды меню

- `1 2 3 ...` — включить/выключить тесты по номерам
- `a` — включить все тесты
- `n` — выключить все тесты
- `p1` — пресет **soak**
- `p2` — пресет **storage-node**
- `p3` — пресет **control-loop**
- `p4` — пресет **full-stack**
- `p5` — пресет **noisy-compute**
- `p6` — пресет **edge-under-io-noise**
- `d <мин>` — задать длительность в минутах
- `h <host>` — задать хост для ping в сетевом тесте
- `z` — включить/выключить CPU noisy-neighbor threads
- `zi` — включить/выключить IO noisy-neighbor thread
- `nt <N>` — задать число CPU noise threads
- `s` — запустить выбранный сценарий/набор тестов
- `q` — выйти из программы

## Пресеты

### `p1` — soak
Включает:
- CPU
- Thermal
- PSI
- hwmon

Основная характеристика:
- **устойчивость вычислений под длительной нагрузкой**

Пресет, чтобы сравнивать:
- sustained CPU throughput
- рост температуры
- поведение при троттлинге
- CPU pressure при длительной вычислительной нагрузке

### `p2` — storage-node
Включает:
- Storage
- Thermal
- PSI
- hwmon

Основная характеристика:
- **отклик накопителя и tail latency**

Пресет, чтобы сравнивать:
- `fsync` p99
- metadata p99
- ухудшение хвостовых задержек со временем

### `p3` — control-loop
Включает:
- Realtime
- Thermal
- hwmon

Основная характеристика:
- **пригодность для soft real-time задач**

Пресет, чтобы сравнивать:
- jitter p95/p99/max
- количество пробуждений выше 500 мкс / 1000 мкс

### `p4` — full-stack
Включает:
- все тесты

Основная характеристика:
- **общий профиль системы**

Пресет, когда нужен один длинный смешанный прогон, но не как единственный режим сравнения плат.

### `p5` — noisy-compute
Включает:
- CPU
- Thermal
- PSI
- Realtime
- hwmon
- CPU noise

Основная характеристика:
- **устойчивость вычислений и таймеров при помехах**

Пресет, чтобы сравнивать:
- деградацию CPU под конкурирующей нагрузкой
- рост jitter под CPU noise

### `p6` — edge-under-io-noise
Включает:
- Storage
- Network
- PSI
- Thermal
- IO noise

Основная характеристика:
- **поведение edge-узла при IO-помехах**

Пресет, чтобы сравнивать:
- рост storage tail latency под шумом
- RTT сети под IO pressure

## Описание тестов

## 1. CPU

### Цель
Понять, может ли плата долго держать вычислительную нагрузку и насколько её производительность деградирует со временем.

### Выходные файлы
- `cpu.csv`
- `cpu_summary.txt`
- `summary.json`

### Основные метрики
- `peak_throughput`
- `sustained_avg_throughput`
- `drift_percent`
- `time_to_first_5pct_drop_sec`
- `time_to_first_10pct_drop_sec`
- `time_to_throttle_sec`
- текущая частота каждого ядра из `/sys/devices/system/cpu/.../scaling_cur_freq`

### Как интерпретировать
Лучшая плата:
- даёт более высокий sustained throughput
- имеет меньший drift_percent
- позже показывает падение на 5% / 10%
- позже уходит в троттлинг или не уходит вообще

Это один из самых сильных тестов для сравнения плат.

## 2. Thermal

### Цель
Оценить тепловой запас платы и рост температуры под нагрузкой.

### Выходные файлы
- `thermal.csv`
- `thermal_summary.txt`
- `summary.json`

### Основные метрики
- `start_temp_c`
- `end_temp_c`
- `max_temp_mc`
- `temp_rise_c_per_min`
- `time_to_critical_sec`

### Как интерпретировать
Лучшая плата:
- медленнее нагревается
- имеет меньшую максимальную температуру при той же нагрузке
- позже достигает критических порогов или не достигает их вообще

Thermal особенно полезен в связке с CPU или Storage-нагрузкой.

## 3. PSI

### Цель
Понять, испытывает ли система pressure по CPU, памяти и IO с точки зрения ядра Linux.

### Выходные файлы
- `psi.csv`
- `summary.json`

### Основные метрики
- `cpu_some_avg10_max`
- `mem_some_avg10_max`
- `io_some_avg10_max`
- delta накопленных pressure-счётчиков по CPU/memory/IO

### Как интерпретировать
Лучшая плата или более удачная конфигурация:
- даёт меньший pressure в одном и том же сценарии

PSI особенно полезен как **объясняющая метрика**: он помогает понять, почему ухудшились throughput или latency.

## 4. Storage

### Цель
Оценить tail latency накопителя и скорость операций с метаданными.

### Выходные файлы
- `storage.csv`
- `mmc_health.txt`, если доступен
- `summary.json`

### Основные метрики
- `fsync_us` p50/p95/p99/max
- `create_us`, `unlink_us`, `mkdir_us`, `rmdir_us`
- `fsync_p99_first_half_us`
- `fsync_p99_second_half_us`
- `fsync_p99_growth_percent`
- `metadata_p99_us`

### Как интерпретировать
Лучшая плата:
- имеет меньший `fsync` p99
- имеет меньший metadata p99
- показывает меньший рост `fsync` p99 по мере длительного прогона

Это один из самых практичных тестов для edge-узлов, локальных БД, логирования и очередей.

## 5. Network

### Цель
Оценить базовую стабильность сетевого RTT лёгким встроенным способом.

### Выходные файлы
- `network.csv`
- `summary.json`

### Основные метрики
- `rtt_ms` p50/p95/p99/max

### Как интерпретировать
Лучшая плата или более удачный сетевой путь:
- имеет меньший RTT
- имеет меньший разброс p95/p99

### Важное замечание
Для серьёзного сравнения разных SBC лучше использовать **локальный peer в той же сети**, а не публичный интернет-хост. Иначе на результат будут влиять провайдер и маршрут.


## 6. cgroup v2

### Цель
Считать статистику ресурсов на уровне текущей cgroup.

### Выходные файлы
- `cgroup.csv`
- `summary.json`

### Основные метрики
- delta CPU usage
- delta throttling
- delta количества throttled periods
- `memory.current`, если доступен

### Как интерпретировать
Сейчас это **вспомогательный телеметрический тест**, а не полноценный изолированный multi-tenant benchmark.

Он полезен для наблюдения за окружением, но noisy-neighbor режим в этой версии реализован через внутренние noise-thread’ы, а не через отдельные cgroup sandbox.

## 7. Realtime

### Цель
Измерить jitter таймеров для soft real-time задач.

### Выходные файлы
- `realtime.csv`
- `summary.json`

### Основные метрики
- jitter p50/p95/p99/max
- `over_500us_count`
- `over_1000us_count`

### Как интерпретировать
Лучшая плата или лучшая конфигурация:
- имеет меньший p99 jitter
- имеет меньший max jitter
- имеет меньше пробуждений выше порогов

Этот тест особенно полезен для control loop, polling-задач, industrial и embedded-сценариев.

## 8. hwmon

### Цель
Снять best-effort аппаратную телеметрию, которую система экспортирует через интерфейс Linux hwmon.

### Выходные файлы
- `hwmon.csv`
- `summary.json`

### Основные метрики
Зависят от платы и драйверов. Типичные датчики:
- температура
- напряжение
- ток
- мощность
- энергия

### Как интерпретировать
Этот тест в первую очередь служит **дополнительной телеметрией**. На одних платах он почти дублирует thermal, на других может дать напряжение, ток или мощность.

## Какие файлы создаёт каждый запуск

Каждый запуск создаёт каталог вида:

bench_results_YYYYMMDD_HHMMSS


Обычно в нём есть:
- `run_info.json` — выбранные тесты и обнаруженные возможности системы
- `system_info.json` — kernel, параметры платы, governor, память, root filesystem
- `summary.json` — основные производные метрики для сравнения
- `DONE.txt` — маркер успешного завершения
- по одному CSV на каждый включённый тест

### Сравнение вычислительной устойчивости
На каждой плате запусти:

p1
d 15
s

Сравни в `summary.json`:
- `cpu.peak_throughput`
- `cpu.sustained_avg_throughput`
- `cpu.drift_percent`
- `cpu.time_to_first_10pct_drop_sec`
- `thermal.max_temp_mc`
- `thermal.temp_rise_c_per_min`

### Сравнение storage

p2
d 10
s

Сравнить:
- `storage.fsync_us.p95`
- `storage.fsync_us.p99`
- `storage.fsync_p99_growth_percent`
- `storage.metadata_p99_us`

### Сравнение control-loop

p3
d 10
s

Сравнить:
- `realtime.jitter_us.p95`
- `realtime.jitter_us.p99`
- `realtime.jitter_us.max`
- `realtime.over_500us_count`
- `realtime.over_1000us_count`

### Сравнение под помехами

p5
d 10
s

и/или

p6
d 10
s

Можно сравнить, насколько CPU, storage и jitter деградируют под шумом.

## Для чего этот бенчмарк особенно полезен

Самые сильные сценарии использования:
- сравнение **устойчивости вычислений под длительной нагрузкой**
- сравнение **теплового поведения под нагрузкой**
- сравнение **storage tail latency**
- сравнение **soft real-time jitter**
- наблюдение за **pressure stalls**, пока работают другие тесты

## Текущие ограничения

- сетевой тест пока в основном основан на `ping` и не является полноценным встроенным throughput-бенчмарком
- cgroup test пока не создаёт отдельные изолированные cgroup автоматически
- storage test специально сделан лёгким и проще, чем `fio`
- realtime test подходит для soft real-time, но не заменяет полноценную RT-валидацию
